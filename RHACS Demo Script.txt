Red Hat Advanced Cluster Security Demo Guide
Built for Version 4.10
About this Guide
This RHACS demo guide is purpose-built for the ACS Demo Environment  It is a living document that will reflect the current state of the product as features are added. It is meant to be a great starting demo flow for field and technical teams to understand the use cases of RHACS.


Feedback is always welcome and appreciated!

You can comment and suggest changes directly in the document or add it to the ISSUES page.
Ordering the demo environment  (Required: Est time ~1 hrs)
1. Log into RHDP. 
2. Search for “Advanced Cluster Security Demo (CNV)” (or use this  link). 
3. Click ‘Order’. 
4. Order page
   1. Fill out the fields appropriate to your use case and make sure to add your case number/oppID
   2. IMPORTANT: Turn off the auto-stop 
5. Click ‘Order’.
6. Wait ~1 hour for the completion email.
7. Access the RHDP platform “Services” tab for the showroom instance and all of the set up details.
Environment setup (Required: Est time ~15 min)
1. Click the Showroom URL and follow the set up instructions.
2. Run the set up commands to ensure all of the use cases are configured in the environment.
3. Run the verify script to ensure everything is set up correctly.
4. Proceed through the demo walkthrough.

UI Overview
Navigate to the Dashboard tab from the left side menu
Dashboard
* The ACS dashboard serves as the security overview - helping the security team understand the sources of risk, categories of violations, and gaps in compliance. 
* All of the elements are clickable for more information, and categories are customizable
Top bar
* Hover over the top bar in the UI
* Near the top, we see that we have one OpenShift cluster with a total of 7 nodes, but the platform supports observation of any number of OpenShift clusters in a single UI.
* And with a click we can get information on the Violations, Deployments and the usage of Images and Secrets in the clusters.
* The top bar also provides links to Search, Command-line tools, Cluster Health, Documentation, API Reference, and logged-in user account
Left menus
* Hover over the left-hand side menu bar
* The left-hand menus provide navigation into each of the core security workflows
*  The left-hand menus also contain the Platform Configuration tab with various Policy Management, Administration Events and integration options.
Global Search
* The global search allows you to search for any data that ACS tracks—vulnerabilities, images, deployments, and runtime—on every page throughout the UI.
* It’s the place to get answers to questions like, “Are we using this image anywhere?” “Are we impacted by log4shell (CVE-2021-44228)?” and " Who’s running user-interactive shell commands in a production container?”
Vulnerability Management
Click on Vulnerability Management on the left-hand side menu to expand it, then on the RESULTS tab
Overview
* Let’s start with Vulnerability Management, a core use case for most security teams.
* The Results page lists all CVEs in all containerized  and OpenShift Virtual Machine workloads running on your clusters. 
* Workloads are separated at the top of the page to help with filtering.
   * User workloads = user applications
   * Platform = OpenShift Platform components
   * Nodes = Node Operating System.
* This helps to diagnose where the vulnerability is originating from and what layer to fix it. 
* By default, ACS filters CVEs to show only Important and Critical vulnerabilities that are "Fixable"—they have a patch available. This particular environment may not show anything with these filters enabled. You can remove these filters to see a complete view of all vulnerabilities.
* In “User Workloads” view, click on the button labeled "<#> Deployments"  
* You can also view vulnerability data organized by Images or Deployments. Image details include CVE counts by severity, Operating System, and the number of Deployments using this image.
* More important than fixing any vulnerability is establishing a process to keep container images updated and prevent the promotion through the pipeline of images with serious, fixable vulnerabilities.
* Other controls on this page allow you to switch to a Namespace-oriented view of vulnerabilities or to view Deferred or False Positive CVEs. CVE reporting workflows are managed under the Exception Management page.

Risk
Navigate to the Risk tab from the left-side menu
Overview
   * Let’s take a look at the Risk view, where we go beyond the basics of vulnerabilities to understand how deployment configuration and runtime activity impact the likelihood of an exploit occurring and how successful those exploits will be.
   * This list view shows all deployments in all clusters and namespaces, ordered by Risk priority.
   * Risk is also influenced by runtime activity - and Deployments that have activity that could indicate a breach in progress have a red dot on the left.
   * Obviously - the first one in the list should be our first focus.
   * The reality of security is that it’s just not possible to tackle all sources of Risk, so organizations end up prioritizing their efforts. 
   * We want RHACS to help inform that prioritization.  
Single Deployment Details
   * Click on the number 1 deployment, “visa-processor” to bring up the Risk Indicators on the right
   * The Risk Indicators tab shows why this deployment is considered a high risk.
   * The deployment has serious, fixable vulnerabilities, but it also has configurations like network ports and service exposure outside the cluster, making it more likely to be attacked.
   * In addition, other configurations, like privileged containers, allow a successful attacker to access the underlying host network and filesystem, including other containers running on that host.
   * Navigate to the bottom of the Risk Indicators page to see the Service Reachability and Components Useful for Attackers sections
   * At the bottom, we see several other Risk factors that ACS interprets automatically. "Service Reachability" is a measure of how exposed the service is. Services exposed within the cluster, or connected to a Route that allows external access, are more likely to be attacked. Components Useful for Attackers are software packages like system utilities, network configuration, shells, and debug or troubleshooting tools that attackers can use once they have a foothold in the container. While these tools aren't malicious, they can be used maliciously to exploit an initial misconfiguration.
   * All of these configurations are gleaned automatically by ACS from OpenShift, and the built-in policies assign a risk score to each, meaning that this Risk report is available as soon as you start running ACS.

Network Policy Simulator
   * Click on Network Policy generator button in the top right
   * The Network Policy Simulator is designed to help solve this problem by using the history of observed traffic to build firewall rules.
   * Click on Generate and Simulate network policies
   * You’ll notice, though, that the firewall rules we’re generating are not proprietary rules, but OpenShift-native Network Policy objects.
   * This feature, more than any other, illustrates the philosophy that ACS represents: Security through platform-native features, with fixes supplied as configuration for OpenShift.
   * By implementing stronger security through declarative code we believe this avoids the anti-pattern of having configuration rules in a separate system - this code becomes part of your application, ensuring the consistency of “single source of truth” from your codebase.
   * This approach also reduces operational risk since there is no proprietary firewall in your cluster or in your pods that could fail, causing an application outage.
   * ACS is leveraging the firewall that’s already in your OpenShift cluster.
   * You’ll see this approach - “fix it in the code,” “leverage the platform,” - everywhere throughout the product - and we’ll take a look at what those policies look like, with examples of policy violations.

Policy Summary
   * ACS has a number of built-in policies to detect activity that is related to attacker goals: gain a foothold, maintain a presence, move laterally, and exfiltrate data.
   * The continuous runtime monitoring observes all container activity and will automatically respond to events with appropriate enforcement and notification
   * In the top menu of the Violation, click on the Policy tab
   * But that would be missing out on an opportunity - ACS wants to go one step further, to take advantage of the ephemeral, immutable nature of containers to improve security in a measurable way going forward.

Policy Management
Navigate to the Policy Management page from the Platform Configuration tab from the left side menu
Overview of Built-in Policies
   * All of the policies that ship with the product are designed to provide targeted remediation that improves security hardening.
   * This list contains many Build and Deploy time policies to catch misconfigurations early in the pipeline, as well as Runtime policies that point back to specific hardening recommendations.
   * These policies come from us at Red Hat - our expertise, our interpretation of industry best practices, and our interpretation of common compliance standards, but you can modify them or create your own.
Specific Policy Example - Alpine Package Manager in Image
   * Click on the Alpine Linux Package Manager (apk) in Image Policy.
   * This is what an ACS policy looks like - front and center, you can see the Rationale and Guidance designed to give the DevOps team some context about why this issue is important for security and, more importantly, what to do about the issue.
   * You might get the impression that we don’t like Package Managers!
   * While useful, containers just represent a tool that an attacker can use to install software and don’t have a legitimate use—a container should ship with all of the dependencies it requires.
Power of a single policy engine
   * Policy criteria can cross the build, deploy, and runtime lifecycles.
   * For example, policies highlighting vulnerabilities in deployments with privileged containers in that deployment.
   * Another example might be runtime criteria - like execution of shell commands - in containers in deployments that have external network exposure.
   * It’s easy to write a policy that prevents use of compilers and other build tools - except in development clusters, in namespaces for CI/CD tools.
   * There are no “silos” - other tools require you to manage policies for vulnerabilities and runtime separately.
   * The unified policy engine allows for targeted conditions and targeted enforcement, easily allowing exceptions for specific applications once approved by security.

   * We’re essentially programming OpenShift to prevent security risk

CI/CD with OpenShift Pipelines


In this section, we will take a look at the broader implementation of security in the development process. We’ve shown how ACS covers the lifecycle of applications from build to deployment to runtime, so let’s dive a little deeper. 
Overview
   * Red Hat® OpenShift Pipelines is a Kubernetes-native CI/CD (Continuous Integration/Continuous Deployment) solution that automates the process of building, testing, and deploying applications. 
   * It leverages Tekton, a powerful and flexible open-source framework, to create cloud-native pipelines that integrate seamlessly with OpenShift and other Kubernetes environments. 
   * This enables developers to define and manage their build and deployment workflows as code, ensuring consistency, scalability, and repeatability across the software development lifecycle.
   * We also have integration with Red Hat Developer Hub to further enhance the security of the coding and build process.
