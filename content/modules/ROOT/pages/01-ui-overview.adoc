= Module 01: UI overview
:source-highlighter: rouge

include::partial$_attributes.adoc[]

This module introduces the Red Hat Advanced Cluster Security {rhacs_version} console. Presenters walk through the PatternFly-updated dashboard, navigation, global search, and the OpenShift console security plugin to establish the full scope of the platform before diving into individual use cases.

*Estimated time*: 8-12 minutes

== Part 1: Dashboard and top bar

=== Know

_Context: Securing containerized workloads at scale requires visibility across the entire environment from a single control plane._

**Business challenge:**

* Security teams managing containerized applications face fragmented tooling: separate scanners, policy engines, and runtime monitors
* Without a unified view, understanding where risk lives across clusters, namespaces, and deployments requires multiple context switches
* Compliance audits and incident response are slowed by the need to correlate data from disconnected systems

**What RHACS provides:**

* A single security control plane spanning the full application lifecycle: build, deploy, and runtime
* Centralized visibility into vulnerabilities, policy violations, network flows, and runtime activity across all connected clusters
* Actionable data organized by risk priority, not raw alert volume

**What is new in RHACS {rhacs_version} for the portal UI:**

* The RHACS portal uses an updated PatternFly interface across views, pages, and menus. Workflows are unchanged; the experience is modernized and more consistent with Red Hat OpenShift.
* Central URLs automatically redirect from `HTTP` to `HTTPS`, aligning with OpenShift platform standards.
* Dashboard policy categories no longer appear duplicated. Casing validation is synchronized between the controller and the UI so you see one clean list.

**Key message for this audience:**

For platform teams, RHACS integrates with OpenShift natively. No agents to manage, no separate infrastructure to run. For security teams, it surfaces Kubernetes-native risk signals that traditional tools miss entirely.

== Part 2: Navigation and global search

=== Know

_Context: The breadth of what RHACS monitors often surprises teams new to the platform. This section establishes the full scope before the detailed use case demos._

**What RHACS tracks across your environment:**

* Vulnerabilities in container images, running deployments, cluster nodes, and (Technology Preview) virtual machines on OpenShift Virtualization
* Deployment configuration risk: privileged containers, exposed ports, missing resource limits
* Network traffic flows between deployments: both observed baseline and anomalous connections
* Runtime process activity inside pods: what is running and what deviates from the established baseline
* Policy violations across build, deploy, and runtime lifecycle stages
* Compliance posture against standards including PCI-DSS, HIPAA, CIS Benchmarks, and NIST SP 800-190

**Navigation changes worth noting in {rhacs_version}:**

* The *Risk* view uses the PatternFly redesign. It remains the place to rank deployments by exploit likelihood and to drill into risk indicators, process discovery, and network activity.
* The legacy *Configuration Management* dashboard and sub-menus are deprecated. Secrets, RBAC, and deployment inventory remain available through APIs and integrated risk and policy workflows rather than a separate configuration dashboard.
* *Platform Configuration* at the bottom of the menu is where you manage policies, integrations, collections, and access control.

**Why global search matters for security and platform teams:**

Security incidents and audits often start with a question: "Are we running this image anywhere?" or "Which deployments are affected by this CVE?" RHACS answers those questions instantly across the entire environment, without requiring manual inventory, scripting, or cluster access.

=== Show

**What I say**:

"Before we go into the individual use cases, I want to show you the left-hand navigation and the global search. This gives you a sense of the full scope of what RHACS monitors, which will make each section more meaningful."

**What I do**:

. Hover over the left-hand navigation menu and walk through the top-level sections:
+
"Each item here represents a core security workflow. We will cover *Risk*, *Vulnerability Management*, *Compliance*, *Violations*, the *Network Graph*, and *Integrations* under *Platform Configuration*. Configuration Management as a standalone dashboard is deprecated in {rhacs_version}; that data now surfaces in Risk and policy workflows."

. Click the *Search* icon in the top bar to open Global Search.

. Type `Process Name` in the search bar and select that key from the suggestions.

. Enter `bash` and press Enter, then click away to clear the dropdown.
+
"There are five deployments that have been seen running bash since they started -- all of them in production. Global search lets your security team answer that question immediately, without SSH access or custom scripting."

. Clear the search and type `CVE-2021-44228`:
+
"Same approach for CVEs. 'Are we impacted by log4shell?' is a one-line query across the entire environment. You get affected images, deployments, and namespaces in a single result."

. Close the search.

. Navigate to *Risk* from the left side menu.
+
"The Risk page was redesigned in PatternFly for {rhacs_version}. This is where RHACS ranks every deployment by likelihood of successful exploit, combining vulnerability, configuration, and runtime signals."

. Point to the *Create Policy* button in the upper right of the Risk page:
+
"Once you have a search or filter that identifies interesting criteria, you can create a policy directly from that context. That policy then runs continuously, automatically identifying this same criteria going forward."

**What they should notice:**

* Search spans all tracked data: CVEs, images, deployments, runtime process activity
* Results are cross-referenced so a CVE query returns affected deployments, not just a finding list
* No query language to learn; search uses structured keys with auto-complete
* The PatternFly Risk view is the hub for prioritizing remediation, not just listing CVEs

[NOTE]
====
**Presenter tip:** For security operations teams, emphasize that global search replaces the need for kubectl commands or node-level access during incident investigations. For platform teams, emphasize that this data is collected automatically by the RHACS sensor with no manual instrumentation required.
====

== Part 3: OpenShift console security plugin

=== Know

_Context: Security teams want findings where developers and platform engineers already work. Switching between the OpenShift console and a separate security portal slows triage and remediation._

**What RHACS {rhacs_version} adds (Technology Preview):**

* A dynamic plugin for the OpenShift web console on secured clusters, delivered with the RHACS Operator
* A *Security* navigation entry with a *Vulnerabilities* view scoped to the selected project or namespace
* A *Security* tab on project, namespace, deployment, and daemonset detail pages with workload vulnerability context
* The same vulnerability data as the RHACS portal, without context switching for namespace-scoped triage

**Lab note:**

This demo environment enables the console plugin during installation. On a fresh {rhacs_version} install, the plugin is enabled by default. After upgrading from an earlier RHACS version, you must enable the *Console plugin* when installing or updating the RHACS Operator.

For more information, see the https://docs.redhat.com/en/documentation/red_hat_advanced_cluster_security_for_kubernetes/4.10/html/configuring/accessing-vulnerability-information-in-web-console[Accessing vulnerability information in the OpenShift web console] documentation.

=== Show

**What I say**:

"RHACS {rhacs_version} also meets your teams in the OpenShift console. Operators and developers can see vulnerability context on the resources they already manage, without opening the RHACS portal."

**What I do**:

. Switch to the *OpenShift Console* tab, or open {openshift_cluster_console_url}[window=blank].

. Log in if prompted using the cluster admin credentials from the lab setup page.

. Select a project that contains demo workloads, such as the application namespace used in later modules.

. In the OpenShift navigation menu, click *Security* > *Vulnerabilities*.
+
"This view is scoped to the project you selected. You can pivot by CVEs, images, or deployments -- the same workload-centric model as the RHACS portal, but inside OpenShift."

. Open a *Deployment* (or *DaemonSet*) in that project and click the *Security* tab.
+
"The Security tab on resource detail pages shows vulnerability information for that specific workload. That is the zero-context-switching story for {rhacs_version}: triage where you deploy, enforce where you operate."

. Return to the RHACS console tab to continue the demo modules.

**What they should notice:**

* Vulnerability data is available in both the RHACS portal and the OpenShift console
* Project-scoped views respect OpenShift RBAC; users only see data for namespaces they can access
* The plugin complements the central RHACS console rather than replacing it

[NOTE]
====
**Presenter tip:** If the *Security* menu or tab is missing, confirm the RHACS Operator *Console plugin* is enabled (`ConsolePlugin` resource in the `rhacs-operator` namespace). The lab `01-verify-rhacs-install.sh` script enables it automatically.
====
